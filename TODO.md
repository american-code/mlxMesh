## Scalability

- **Coordinator HA** — One coordinator per region with no failover or shared state. Still not started — this item is leader election, cross-coordinator request routing, and failover, none of which are touched by the ledger work below. That work makes multi-coordinator write-sharing *safe*; it doesn't make coordinators *aware of each other*.
- ✅ **Ledger beyond SQLite** — Investigated `internal/settlement/ledger.go` directly first: it's a concrete struct with no storage interface, and — the important finding — correctness today is enforced entirely by a single in-process `sync.RWMutex`, not by SQLite itself (no transactions, no row locks, no WAL tuning anywhere). That meant a naive fix (just pointing `sql.Open` at Postgres instead of SQLite) would not have solved "won't survive multi-coordinator write load" at all: two coordinator processes each have their own independent mutex, so nothing would have stopped them racing on a balance or double-claiming a startup grant. Built the real fix instead: a new `internal/settlement/ledger_postgres.go` backend (`NewPostgresLedger(dsn)`, `github.com/jackc/pgx/v5/stdlib` driver) where correctness is enforced by Postgres itself — `SELECT ... FOR UPDATE` row-locks a user's running-balance row (`account_balances` table) before `DebitAccount` allows a spend, and a unique constraint on a new `startup_grant_claims` table makes `ClaimStartupGrantOnce` dedup safely across processes via `INSERT ... ON CONFLICT DO NOTHING` rather than a Go-side scan. `Reconcile()`'s Postgres path deliberately re-derives totals from the raw `credit_entries`/`ledger_debits` audit tables (not from the maintained `account_balances`) so it still catches the class of bug it exists to catch. `Ledger`'s public method surface is unchanged — zero call-site changes anywhere in `cmd/coordinator/main.go` — via a `backend` discriminator and a one-line dispatch at the top of each method; the existing SQLite/memory path is untouched and all its tests still pass as-is. New coordinator flag `--ledger-db-url` (Postgres DSN) takes priority over `--db-path` for the ledger specifically; `--db-path` keeps backing the API-key store either way. Verified for real, not just in unit tests: spun up a disposable Postgres (Docker), ran the new `internal/settlement/ledger_postgres_test.go` suite against it (gated on `TEST_POSTGRES_DSN`, skips cleanly otherwise) including two concurrency-proof tests (N goroutines racing `ClaimStartupGrantOnce` → exactly one succeeds; N goroutines racing `DebitAccount` past a fixed balance → never overdraws) — both pass clean under `-race` too. Then went further and proved it across real separate OS processes, not just goroutines: built and ran two actual `oim-coordinator` binaries pointed at the same Postgres instance and fired a concurrent startup-grant claim for the same user at both simultaneously — exactly one succeeded, both processes agreed on the resulting balance afterward. Also drove a full paid job (grant claim → chat completion → consumer debited, node earned 75%, treasury 25%) through a single Postgres-backed coordinator and confirmed the balance survived a coordinator restart. **Deliberately out of scope**: `internal/federation.Store` and the coordinator's SQLite-backed `apiKeyStore` use the identical pattern but aren't Postgres-backed by this change — a scoping decision, not an oversight, since the TODO item named the ledger specifically. This item is the concurrency-safety prerequisite Coordinator HA needs; it does not implement HA itself.

## Economic Sustainability

- ✅ **Bootstrapping economics problem** — Investigated `internal/economics/pricing.go`, `internal/coordinator/availability.go`, and the fast/background-lane billing paths in `cmd/coordinator/main.go` directly first. Key finding: the "serve 33% more tokens than consumed to break even" math is a structural property of the 25% house edge, true in ANY network state — what's actually inactivity-dependent is whether a participant can find enough real demand (consumer traffic, or encrypted-pointer traffic for coordination rewards) to earn from at all; in a truly quiet network there isn't any. The one existing mechanism that doesn't need reciprocal demand is the verified-availability reward (`--availability-reward`, opt-in) — the coordinator manufactures its own probe job and mints a reward directly, gated only by queue backpressure. Shipped two changes, both keyed off `JobQueue.BackpressurePct()` (the same signal already used for exactly this "is the network busy" gating):
  1. **`coordinator.ScaledProbeBudget`** — the availability-reward probe now probes more idle nodes per round the quieter the network is (from the existing floor of 3/round up to a new ceiling of 15/round at 0% backpressure), tapering back down as real traffic returns. Directly implements "minimum guaranteed earnings for active nodes even when idle," scaled to actual inactivity.
  2. **`economics.ConsumerCostWithActivityDiscount`** — fast-lane consumer cost is discounted during quiet periods, down to exactly the provider's own reward share at a fully idle network (0% backpressure) — i.e. the treasury's margin compresses to zero, never negative, while **the provider's payout is completely unaffected** (still computed from the full undiscounted cost). Directly implements "dynamic base cost based on network activity" without touching provider economics.
  - **Caught and fixed a real solvency bug during implementation**: `creditNodeEarning` originally computed the treasury's margin from the undiscounted `economics.NetworkMargin`, independent of what the consumer's (now possibly discounted) debit actually was — meaning the network would have minted more credit than the consumer paid. Fixed by threading the actual `consumerCharge` into `creditNodeEarning` and deriving margin as `consumerCharge - reward` directly, so `reward + margin` always equals exactly what was charged. Caught via a real end-to-end manual verification (not just unit tests) before shipping — see the verification note below.
  - **Explicitly not done, on purpose**: raising `CoordinationRewardPerPointer` when inactive (it's the one thing that actually debits the treasury, and SLOS.md already documents "reduce it" as the prescribed remedy under treasury pressure — raising it during inactivity would fight that same constraint, and it's demand-gated anyway, so there's nothing to reward in a truly dead network); a separate "consumer-only tier" concept (the activity discount already delivers the same relief); changing `--availability-reward`'s default from off (a deployment-behavior/minting-externality decision, not this task's call).
  - Tests: `internal/economics/pricing_test.go` (discount bounds/monotonicity, the "never undercuts provider reward" invariant, the zero-margin-at-fully-idle anchor, "provider reward never moves" regression), `internal/coordinator/availability_test.go` (`ScaledProbeBudget` bounds + degenerate-param fallback), `cmd/coordinator/creditnodeearning_test.go` (new file — the solvency-fix regression tests). Updated `tests/integration_test.go`'s `TestIntegrationFullMoneyPath`/`TestIntegrationStreamingFastLane` to assert the solvency invariant (miner+treasury credited == consumer debited) instead of a fixed ~25% ratio, since a fresh single-node test harness is naturally at 0% backpressure — exactly the case this fix targets. Verified end-to-end against a real coordinator+stub-exo+node: at 0% backpressure, consumer was debited exactly the miner's reward (treasury margin = 0), and the miner's own earning was unchanged from what it would be at full price.

## Performance Optimizations

- 🔶 **Speculative decoding on node side** — **Blocked on upstream Exo, not this repo.** Researched against exo-explore/exo's actual API (`docs/api.md`): `/v1/chat/completions` and `/instance` accept no draft-model/speculative-decoding parameter today — only the underlying `mlx-lm` CLI (`mlx_lm.generate --draft-model ... --num-draft-tokens ...`) supports it, and this repo has zero inference code of its own (`internal/exoadapter` is a pure HTTP client; everything token-level lives in Exo). So the original "drop-in, no protocol changes needed" framing doesn't hold as written. What *is* shippable now: forward-compatible pass-through plumbing, landed this session — `nodeconfig.DraftModelConfig` (`draft_models` in `~/.config/oim/config.json`, keyed by served model_id) flows through `agent.Config` → `capability.Options.DraftModels` (surfaced read-only on `ModelCapability.DraftModelID`/`NumDraftTokens` for dashboard/coordinator visibility) and separately into `jobrunner.Runner.draftExtra`, which merges `draft_model`/`num_draft_tokens` into the outgoing Exo request via the existing opaque `extra` passthrough (`exoadapter.RunChatCompletion`/`StreamChatCompletion`). Configuring this today is inert — Exo silently ignores the unrecognized fields — but the mesh is instantly ready the moment Exo's API (or an operator's fork) adds support, with zero further protocol changes. Covered by unit tests in `internal/nodeconfig/validate_test.go` and `internal/jobrunner/runner_test.go`. **Remaining:** either Exo upstream adds this to its HTTP API, or an operator forks Exo to add it — outside this repo's control either way.
- ✅ **Prefix/KV-cache-aware routing** — `internal/coordinator/consistenthash.go` adds a minimal consistent-hash ring (512 virtual points/member, rebuilt fresh per dispatch from the current eligible candidates — no persisted ring state to keep in sync with churn). `router.go`'s `promptAffinityKey` fingerprints a job's system-prompt content (falling back to the first message when there's none) plus model ID; `rankCandidates` promotes the ring-preferred node to primary whenever it's eligible and not already overloaded (`affinityMaxInFlight`), bypassing the power-of-two-choices stage for that request — deliberately, since concentrating a *shared* prefix onto one warm node is the point, not something to spread away from. Falls back to ordinary power-of-two selection when there's no prefix, no preferred node, or the preferred node is too busy. Covered by 8 unit tests (`router_test.go`) + 3 hash-ring property tests (`consistenthash_test.go`) + a real end-to-end integration test (`TestIntegrationPrefixAffinityKeepsRepeatedPromptsOnTheSameNode`) proving repeated identical prompts stay pinned to one real node while varied prompts spread across both. **Flakiness fix (post-ship):** the integration test's "spread across both nodes" check failed ~1 in 3 runs — root-caused via simulation to real per-instantiation skew in a 2-member/32-replica ring (not a routing bug: a random 2-member ring is >70/30-skewed ~20% of the time with only 32 replicas, occasionally much worse, and only 12 sample prompts isn't enough to reliably reveal a spread even under a correctly-working ring). Fixed by raising `hashRingReplicas` to 512 (empirically 0/5000 simulated failures, vs 2/5000 at 256 and ~40% at 32 — construction cost is still microseconds, so this is a pure quality win for small fleets in production too) and raising the test's distinct-prompt probe budget to up to 200 (early-exits the moment both nodes are seen, so the common case is unaffected). 30 consecutive re-runs passed after the fix.
- ✅ **Power-of-two-choices for fast-lane dispatch** — `internal/coordinator/router.go`'s `rankCandidates` now samples 2 of the top-`powerOfTwoWindow` (3) score-sorted candidates and promotes the better-scored to primary, instead of always the single global argmax — breaks the herding failure mode (concurrent requests evaluated against the same registry snapshot no longer all agree on one target before in-flight load can react) while bounding quality loss to "the window's weakest member," not a random node from the whole fleet. The deterministic best-first order is kept for the retry/fallback tail, since a dispatch *failure* (not load) is what fallback recovers from. Covered by 5 new unit tests in `router_test.go` (herding-fix variance, window bound, fallback-order stability).
- **Kademlia DHT for M7 directory federation** — `FederatedResolver`/`DHTResolver` are stubbed. Kademlia enables efficient node discovery and data lookup in decentralized P2P networks using XOR distance, O(log n) messages to locate any key/node, highly resilient to churn. Solves discovery, not ledger consensus (those stay separate concerns).

Unit + live end-to-end tests (real coordinator/stub-exo/node binaries) in `python-sdk/tests/`. **Remaining (operator):** reserve the `mlxmesh` name and actually publish to PyPI — needs an account/credentials, not engineering.
- ✅ **Swift SDK** — `swift-sdk/` (SwiftPM library `MeshKit`, iOS 16+/macOS 13+/tvOS 16+/watchOS 9+). Same surface as the Python SDK (`MeshClient` actor, `MeshDirectory`), plus `PayloadEncryption` adapted from `OIMDashboard/iOS/Crypto/PayloadEncryption.swift`. Its cross-language interop test caught a real, previously-unknown bug in that very file — CryptoKit's `.rawRepresentation` for the ephemeral P-256 key is the compact 64-byte form, not the 65-byte SEC1 form Go's `crypto/ecdh` requires — silently breaking every real encrypted-pointer job end to end; fixed in both the new SDK and the original dashboard file. `swift test` passes; confirmed `OIMDashboard`/`OIMMenuBar` still build clean. **Remaining (operator):** publish via Swift Package Manager (tag a release) — needs repo/release setup, not engineering.
- ✅ **Wallets + enforced payment for both SDKs** — Prior to this, neither SDK had any wallet/identity code: `MeshClient` took only opaque caller-supplied `api_key`/`user_id` strings, with **no client-side check that a credential existed at all** before firing a billable request. Added a new `Wallet` type to each SDK (`python-sdk/src/mlxmesh/wallet.py`, `swift-sdk/Sources/MeshKit/Wallet.swift`): generates an Ed25519 keypair locally, derives the address exactly as `internal/wallet.AddressFromPubKey` does (`"oim" + hex(sha256(pubkey))`), signs the real `POST /account/challenge`/`POST /account/auth` round trip, and persists to disk (0600 perms) — verified byte-exact interop against the actual Go `wallet.Manager` before writing any SDK code. `MeshClient`/`MeshKit.MeshClient` now **refuse outright, before any HTTP request**, when a billable call (`chat`/`stream_chat`/`submit_background_job`/`run_background_cycle`/`reserve_node`/`submit_encrypted`) has neither a wallet nor a pre-existing `api_key` configured (`NoCredentialError` / `MeshClientError.noCredentialConfigured`) — the concrete fix for "don't work if a user can submit jobs without paying." A wallet mints its own credential transparently on first use and self-heals once on any 401. The pre-existing static-key workflow is unaffected (dedicated regression test in both SDKs). Verified against a coordinator with `--api-key` actually enforced (not the old wide-open dev-mode harness) — both SDKs' full test suites green (41 Python, 37 Swift).
