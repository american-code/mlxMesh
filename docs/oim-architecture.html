Open Inference Mesh — Architecture & Data Reference

Generated from a direct code review of github.com/open-inference-mesh/oim (Go coordinator/directory/node stack + the OIMDashboard iOS/watchOS/tvOS app). Diagrams render with Mermaid — open this file in a browser to view them live.

Three binaries, one hierarchy: oim-directory (global, discovery-only, never touches a job payload), oim-coordinator (one per region/pod — owns routing, settlement, security), and oim node (wraps a local Exo instance and does the actual inference work). Everything above the node layer is coordination and accounting; Exo is treated as a black box.

1. How does the network work?

The mesh is a strict three-layer hierarchy. The Global Directory (oim-directory, port 9100) is a pure discovery service — pods register a signed, aggregate-only health digest with it and clients ask it "who serves model X," but it never sits in the inference path and never sees a job payload. Pod Coordinators (oim-coordinator, one per region) are where the real logic lives: node registry, fast/background-lane routing, settlement, wallet challenges, attestation checks, and federation. Node Agents (oim node start) each wrap one local Exo instance and do nothing except serve jobs Exo can run.

Node registration

A node boots, loads or creates an Ed25519 identity at ~/.config/oim/node_identity.json, assembles a CapabilityManifest from live Exo state plus its own resource governor, signs it, and POSTs it to /nodes/register. The coordinator independently recomputes node_id = SHA-256(pubkey)[:32] and rejects the registration if it doesn't match what the node claimed — identity can never be operator-chosen or hijacked. Every RefreshInterval (default 30s) the node re-derives its manifest from live state and re-signs a refresh; capability is never cached across heartbeats.

Exo integration

internal/exoadapter is explicitly "a thin HTTP client wrapping a LOCAL Exo instance... no inference logic lives here." It calls Exo's real endpoints (/state, /models, /instance, /v1/chat/completions) at http://localhost:52415 by default. Multi-device Exo clusters (Thunderbolt-linked Macs) are detected and folded into one mesh node rather than being exposed as separate nodes, with a per-device memory reserve (2 GB or 15%, whichever is larger) that a node keeps back regardless of what the operator configures as its contribution cap.

MoE expert-sharding, not dense pipeline-sharding

CLAUDE.md's core constraint — no WAN pipeline-sharding of dense models, because 20–150ms inter-hop latency kills sequential token passing — is enforced in code, not just documented. internal/coordinator/moe_planner.go assigns Mixture-of-Experts shards to nodes proportional to memory capacity (largest-remainder method, so counts always sum exactly) and can route a token to whichever node holds the activated expert. Notably, none of this MoE machinery is invoked anywhere in the live fast-lane chat path today — it's wired for the background/decomposition lane only, architecturally separate from a single chat request.

Federation — more built than the milestone label suggests

CLAUDE.md lists "M7 stub — federated directory," which undersells what's there. internal/federation is a complete, tested, cryptographically-verified system where pod coordinators witness each other's signed credit-issuance history: every credit event is Ed25519-signed and sequence-numbered per pod, peers poll each other's new events every 30 seconds, and a coordinator that later reports a balance contradicting its own signed history gets caught (fork/impersonation detection, TOFU key pinning). What's genuinely still a stub is federated discovery — a second directory implementation that would let multiple librarians share the pod registry — which is a different package (internal/directory's FederatedResolver) that happens to share the "M7" label.

2. How is it secured?

Security is layered rather than a single mechanism: identity, transport, request signing, payload confidentiality, an attestation gate for sensitive jobs, anti-abuse at the HTTP perimeter, and proof-of-work against economic Sybil attacks.

LayerMechanism
Node identityEd25519 keypair, node_id = SHA-256(pubkey)[:32] hex, recomputed and checked by the coordinator on every registration — never trusts an operator-supplied ID.
TransportTLS 1.2+ where certs are configured; for independently-operated, self-signed nodes, TOFU certificate-fingerprint pinning substitutes for CA trust, captured once at registration.
Request signingEvery state-mutating node→coordinator call (refresh, benchmark submission, job outcome, enclave attestation) is Ed25519-signed and timestamp-bounded — rejected if older than 5 minutes or more than 30 seconds in the future. Verification always uses the public key captured at registration, never one supplied in the request.
Payload confidentialityECDH(P-256) → HKDF-SHA256 → AES-256-GCM, implemented byte-identically in Go (internal/payloadcrypto) and Swift (PayloadEncryption.swift). The coordinator only ever routes an encrypted pointer (content hash + fetch URL + ephemeral public key) — it never sees a raw prompt. Only the node holding the matching ECDH private key can decrypt.
Sensitivity tierslow / moderate / high_requires_attestation. The high tier requires a coordinator-verified Apple Secure Enclave challenge — a dual signature (fresh ephemeral P-256 key from the enclave + the node's already-registered Ed25519 key) proves both "this hardware has an enclave" and "the node claiming it owns the registered identity." A client-side sensitivity hint can only escalate a job's tier, never de-escalate or bypass the gate.
Perimeter hardeningToken-bucket rate limiting (by IP or user), 8 MiB request body cap, concurrency semaphore with immediate 503 rather than unbounded queueing, SSRF-hardened URL fetch (blocks loopback/link-local/cloud-metadata addresses but allows private LAN ranges since legitimate iOS pointer hosts live there), redirect and dial-time IP re-validation to close DNS-rebinding gaps, and standard security response headers.
Anti-Sybil economicsStartup-grant claims require proof-of-work (18 leading zero bits on SHA-256(userID‖nonce), roughly a quarter-million hashes — sub-second on real hardware but expensive to farm at scale), plus a per-IP hourly rate limit and a one-time-claim guarantee in the ledger.
Gap worth knowing about: the coordinator's probabilistic spot-check re-dispatch and 3σ statistical baseline drift detection exist as tested functions but are not currently called anywhere in the live /v1/chat/completions handler — fast-lane responses aren't spot-checked or baseline-checked today, despite CLAUDE.md describing that verification layer as part of the live milestone. The only verification actually exercised per-node is the separate, node-initiated tier-claim check.

3. How does a message flow through the mesh, from a user and back?

Tracing a single fast-lane chat completion end to end (the interactive path — a background/batch job instead goes through a scheduler-routed, sticky-session path with mandatory recurrence, which is architecturally separate and never shares fast-lane code):

    U --> V["Credit serving node/user earned balance"]

    V --> X["200 OK — OpenAI-shaped JSON, tagged with serving node, latency, tok/s"]

A few details that matter: routing never trusts a node's self-declared specs — it scores candidates using the last measured benchmark signature, penalized by current in-flight load. Failover marks a failed node unreachable and tries the next-ranked candidate (up to a configurable attempt cap) rather than retrying the same node. If every attempt fails and the client opted in via a header, the job falls into a short-lived bounded queue instead of failing outright. Streaming works identically through the routing/scoring stage, but relays Server-Sent Events end to end through the same relay helper at all three hops (Exo → node → coordinator → client); a stream that dies mid-flight can't be failed over — the client already has partial output — so it's logged rather than retried. Billing and node payout are always based on tokens the coordinator itself observed in the response, never a number the node reports about itself.

4. How do regions scale and activate?

A new region comes online by starting an oim-coordinator pointed at the global directory with a pod ID and region hint. From that moment it immediately, and then on a fixed interval (60s by default, 15s in the Docker simulation), builds a signed aggregate-only health digest from its live node registry — model list, node count, and summed measured throughput, never any per-node detail — and pushes it to every configured directory endpoint.

The directory verifies the digest's signature against a pinned public key (trust-on-first-use by default, or a strict pre-authorized allowlist), stores it in a 120-second TTL cache, and forwards it one hop to a single paired peer directory for bootstrap redundancy — deliberately not full multi-hop gossip, to avoid loop amplification. Clients and dashboards discover pods by querying the directory (GET /pods, GET /topology) and then talk to the resolved pod's coordinator directly for everything else; the directory is never back in the loop for an actual job.

New users joining a pod get a startup grant, but the grant amount is a discrete step function of that pod's verified capacity — throughput from nodes whose self-submitted benchmark actually passed comparison against their claimed signature, not raw registered numbers. Below a verified-capacity threshold a pod grants the full base amount; past a middle threshold it halves; past a high threshold it grants nothing — directly implementing "bootstrap grants decay as earned capacity grows" without letting someone spin up fake unverified nodes to keep grants flowing. Grant issuance is itself gated by proof-of-work and a per-IP hourly cap to prevent farming.

Cross-region settlement integrity rides on the federation layer described above: coordinators poll each other's signed ledger-event history and flag any pod whose reported balance contradicts what it previously signed, so a compromised or forked coordinator's claims are independently checkable by every peer that's witnessed its history.

Concretely: a node's CapabilityManifest captures declared memory and contribution cap, model list and quantizations, geographic hint (coarse lat/lng), a reachability endpoint, price per unit, whether it self-declares a Secure Enclave, and its last measured benchmark signature. A job (JobSpec) captures who requested it, which lane and sensitivity tier it ran under, and its price ceiling. Every unit of work actually delivered is itemized into resource lines (memory-hours, compute cycles, bandwidth relayed) with both the measured and delivered amounts recorded — the difference is tracked explicitly as shrinkage, not silently dropped. Those lines roll up into a division order per job/node pair, which gets embedded in an Ed25519-signed settlement record.

The credit ledger is the one place CLAUDE.md's "division-order accounting, never one collapsed number" constraint is most visible in code: a user's balance is always two independently-summed numbers — grant credits and earned credits — combined into a total only for display, and debits are applied grant-first. A self-audit function independently re-derives that same split and flags any account where debits exceed credits. Job outcomes, benchmark results, and enclave attestations are all signed submissions tied back to a node ID, and a separate measurement store tracks a node's most recently measured signature apart from what it originally claimed at registration — the gap between the two is what tier-verification checks.

6. How do iOS devices add to the network?

Short answer: iOS devices never run inference and never register in the node registry — that's true by explicit design in both the Swift and Go code, not an oversight. Exo, the mesh's inference backend, is a macOS/Linux-only MLX daemon; there is no CoreML-as-inference-server path, no MLX/llama.cpp binding, and no HTTP server on the iOS side that executes a model.

What the iOS app (OIMDashboard, with iPadOS/visionOS support) actually does is threefold:

RoleWhat it does
DashboardReads live mesh state — topology, node list, region/pod summaries, global map — the same read-only surface watchOS and tvOS expose exclusively.
ClientSubmits real, metered POST /v1/chat/completions calls against the mesh ("Try the mesh"), backed by a portable wallet: an Ed25519 account keypair held in iCloud Keychain, whose address ("oim" + hex(SHA-256(pubkey))) is the same key the coordinator's ledger balances are stored under — recoverable by re-importing the key, with no server-side reset since the coordinator never holds it.
Optional coordination participantA device can opt in to a lightweight "Contribute" session that runs a small on-device intent classifier (CPU/Neural Engine only, never GPU) to hint a job's routing/sensitivity, and can host an encrypted payload pointer — using the byte-identical ECDH/HKDF/AES-GCM scheme as the Go side — so a real compute node can fetch and decrypt a prompt without the coordinator ever seeing plaintext. This is tracked in a separate CoordinationRegistry with its own TTL, explicitly walled off so it can never be selected by the inference routers, and gated by live thermal and memory-pressure checks so it never competes with the phone's foreground use.

watchOS and tvOS are strictly passive — polling the directory's topology/node endpoints with no wallet, no crypto, and no coordination code present in those targets at all.

Appendix: where this report corrects CLAUDE.md

Three places where the project's own CLAUDE.md undersells or mislabels what's actually in the code, found while researching this document:

"M7 stub — federated directory" is misleading. Cross-pod signed ledger-event witnessing and fork/impersonation detection (internal/federation) is complete, tested (8 passing tests), and wired live into credit issuance with three working HTTP endpoints and a background poller. What's genuinely still unbuilt is a second, federated discovery layer (multiple directory "librarians" sharing a pod registry) — a different package that happens to share the M7 label.
M3's fast-lane verification (probabilistic spot-check re-dispatch, 3σ statistical baseline drift) is listed as live coordinator functionality but isn't called anywhere in the current /v1/chat/completions handler. It exists as tested code, just not wired into the request path yet.
The milestone table doesn't mention several substantial packages that exist and work — wallet identity, payload encryption, Secure Enclave attestation, HTTP-perimeter hardening (rate limiting, SSRF protection), and the iOS app's full crypto/wallet/coordination stack — which makes the written milestone list read as less complete than the codebase actually is.
